The scanner noise problem

A typical mid-size org runs 5–15 scanners. Each produces hundreds of findings per scan. A critical RCE in a public-facing payment service sits next to a low-severity typo in an internal tool — both labeled "High."

How it works

⬇

Ingest

POST findings from any scanner — Snyk, Trivy, Grype, Semgrep, Checkov, or custom tools. Two endpoints: structured schema and flexible webhook.

POST /webhook/findings
→
⚡

Score

A deterministic rule engine assigns a score (0–140) using environment, exposure, data sensitivity, and business context. No black boxes.

production +40 · internet_exposed +30
→
🔗

Correlate

5 built-in rules detect clusters — same-service multi-findings, CVE reuse, exposure+sensitive combinations across scanners.

correlate_as_map(findings)
→
🤖

Enrich

LLM analyzes the top findings and returns exploitability context, specific fix, urgency classification, and adjusted priority.

urgency: now | today | this-week
→
📣

Deliver

Ranked top 5 in the dashboard. Slack cards with owner, score, and remediation step. Scheduled via cron or triggered per CI run.

POST /triage/analyze

Deterministic scoring model

Every factor is additive, auditable, and customizable. No weights. No ML. Maximum score: 140.

FactorConditionPoints
Severitycritical+30
Severityhigh+20
Severitymedium+10
Severitylow+2
Environmentproduction+40
Exposureinternet_exposed = true+30
Data sensitivitysensitive_data = true+20
Business criticalityhigh (context map)+20
Business criticalitymedium (context map)+10
Public facingpublic_facing = true (not already internet_exposed)+15

Architecture

API
FastAPI :8000

Ingest, normalize, triage, LLM orchestration. Stateless — scale horizontally.

Worker
Python BRPOP

Async rule engine scorer. Pulls jobs from Redis, updates scores in Postgres.

UI
Streamlit :8501

Dashboard — ingest form, triage view, LLM analyze, health status.

Storage
PostgreSQL 16

Findings and scan runs. Swap for RDS/Cloud SQL in production.

Queue + Cache
Redis 7

Scoring job queue + LLM response cache (24h TTL, SHA-256 keyed).

LLM
Pluggable

OpenRouter · OpenAI · Anthropic. Switch via env var. No code change.

Enterprise integration

1. Push from any scanner

curl -X POST https://trte.your-org.com/webhook/findings \
  -H "Content-Type: application/json" \
  -d '{
    "scanner": "trivy",
    "findings": [{
      "id": "trivy-CVE-2024-5678",
      "service": "payment-api",
      "severity": "CRITICAL",
      "environment": "prod",
      "cve": "CVE-2024-5678",
      "description": "OpenSSL heap buffer overflow"
    }]
  }'

TRTE normalizes CRITICAL→critical, prod→production automatically.

2. GitHub Actions (post-scan)

- name: Push to TRTE
  run: |
    python scripts/trivy_to_trte.py results.json | \
      curl -X POST $TRTE_URL/webhook/findings \
           -H "Content-Type: application/json" \
           -d @-

Works with GitLab CI, Jenkins, CircleCI, Buildkite — any tool that can run curl.

3. Service context map

{
  "payment-api": {
    "criticality": "high",
    "public_facing": true,
    "owner": "payments-team"
  },
  "internal-admin": {
    "criticality": "low",
    "public_facing": false,
    "owner": "platform-eng"
  }
}

Auto-enriches findings at ingest. Generate from Backstage, PagerDuty, or ServiceNow.

4. Scheduled Slack triage

# cron: every morning at 8am
0 8 * * * curl -X POST \
  https://trte.your-org.com/triage/analyze

# Kubernetes CronJob also supported

Posts top 3 risk cards to your #security-triage channel with owner, score, and fix.
